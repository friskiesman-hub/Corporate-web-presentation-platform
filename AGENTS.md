# AGENTS.md

Этот файл содержит обязательные инструкции для Codex при работе в этом репозитории.

## 1. Язык взаимодействия

- Отвечай пользователю на русском языке, если пользователь явно не попросил другой язык.
- Английский сохраняй только для точных технических идентификаторов: file names, folder names, component names, TypeScript types, functions, CSS classes, package names, Git commands, npm/pnpm scripts, config keys, MCP-серверов, Codex skills и официальных названий инструментов вроде `Vite`, `React`, `TypeScript`, `21st.dev Magic MCP`.
- Не используй английские смысловые слова в русских объяснениях, если есть точный и естественный русский эквивалент.
  - `section` -> `секция`, кроме имени типа или компонента;
  - `proof` -> `доказательство`;
  - `evidence` -> `подтверждение данными` или `доказательная база`;
  - `management implication` -> `управленческий вывод`;
  - `baseline` -> `база сравнения`;
  - `current value` -> `текущее значение`;
  - `safe implementation prompt` -> `безопасный промпт на реализацию`;
  - `visual polish` -> `визуальная доводка`;
  - `creative direction` -> `творческое направление`.
- В коде используй английские имена сущностей. Пояснения пользователю — на русском.
- Если в проекте есть англоязычная документация библиотеки или инструмента, не переводи технические API и команды.
- Technical identifiers всегда остаются на английском: file names, folder names, components, TypeScript types, functions, CSS classes, package names, Git commands, scripts и config keys.
- Названия компонентов и типов остаются на английском: `ComparisonEvidence`, `PresentationSection`, `ExecutiveCover`, `KpiStrip`, `PresentationConfig`, `BaseSection`.
- В документации и промптах можно использовать английское имя сущности в backticks, а затем объяснять смысл по-русски.
- Не смешивай русский язык в identifiers, file names, component names, scripts и config keys.
- Если добавляешь новые содержательные комментарии в код, делай их bilingual:
  - English first;
  - Russian second;
  - комментарий должен быть коротким и полезным;
  - не комментируй очевидный код.
- Commit subject должен быть на английском в Conventional Commits style, например `docs: update tooling notes`.
- Не делай commit subject bilingual: это зашумляет `git log --oneline`.
- Для маленьких routine commits достаточно English-only subject без body.
- Commit body должен быть bilingual, если изменение важное, архитектурное или может быть непонятно только из subject:
  - `EN:` concise technical summary;
  - `RU:` краткое объяснение смысла для пользователя/команды.
- Документация может быть Russian-first, если она описывает project guidance, business context, brand logic или user workflow. Technical terms внутри документации сохраняй на английском, когда это повышает точность.


## 2. Управление инструкциями проекта

Файлы инструкций (`AGENTS.md`, `PROJECT_CONTEXT.md`, `QA_CHECKLIST.md`, `DEPLOY.md`, `docs/*.md`, `.agents/skills/*/SKILL.md`) нельзя создавать, переписывать или дополнять по собственной инициативе.

Codex может менять такие файлы только если пользователь явно поставил задачу обновить инструкции, документацию, QA-правила или skills.

Если какой-то файл упомянут в README или документации, но отсутствует в проекте, не создавай его автоматически. Сначала сообщи пользователю о несоответствии и предложи варианты: удалить ссылку, добавить файл или скорректировать структуру.

Файлы с пользовательскими промптами для Codex UI не должны храниться внутри проекта, если пользователь не попросил обратное. Это личная шпаргалка пользователя, а не project instruction.

## 3. Идентичность проекта

Этот репозиторий — новый экспериментальный проект для создания платформы премиальных корпоративных web-презентаций в фирменном стиле BORK.

Главная цель — не сделать одну презентацию, а создать основу системы, которая позволит быстро генерировать сильные, современные и визуально безупречные web-презентации на основе бизнес-данных.

Текущий фокус тем:

- Internet Boutique;
- e-commerce продажи;
- performance marketing;
- digital channels;
- персональные консультанты и продажи;
- CRM и клиентская коммуникация;
- категории и ассортимент;
- доставка, оплата, конверсия, выполнение заказов;
- управленческие отчеты для senior-аудитории.

## 4. Отношение к предыдущей стабильной презентации

Предыдущая стабильная web-презентация BORK Internet Boutique Q2 2026 является visual reference и stable demo.

Ее можно использовать как источник:

- ощущения премиальности;
- цветовой палитры;
- примеров использования лого, паттерна и декоративных бренд-элементов;
- подхода к desktop/mobile адаптации;
- опыта создания кинематографичной бизнес-презентации.

Но она не является архитектурным ограничением.

Не копируй из старого проекта как обязательные правила:

- ровно 5 слайдов;
- plain HTML/CSS/JS;
- старую навигацию;
- текущую структуру секций;
- текущие layout-паттерны;
- текущий уровень сдержанности в анимации;
- текущие классы и техническую архитектуру.

Новый проект должен сохранить brand DNA, но иметь свободу для более смелого дизайна, анимации, инфографики и интерактивности.

## 5. Product ambition

Проект должен привести к платформе, которая способна заменить или превзойти PowerPoint/PDF для внутренних управленческих презентаций.

Результат должен восприниматься как:

- точный;
- дорогой;
- современный;
- управленчески убедительный;
- кинематографичный;
- технологичный;
- эстетически связанный с BORK;
- одинаково качественный на desktop и iPhone Pro Max.

Презентация должна превращать бизнес-данные в ясную управленческую историю, а не просто показывать набор красивых экранов.

## 6. Что это за проект

Проект является:

- creative lab для premium web-presentations;
- будущей системой генерации корпоративных презентаций;
- библиотекой reusable presentation sections;
- системой brand tokens, layout patterns и motion principles;
- средой для экспериментов с animated infographics;
- системой превращения business data в executive storytelling.

Проект может содержать:

- demo presentations;
- prototypes;
- reusable components;
- design tokens;
- mock data;
- content schemas;
- reference assets;
- Codex skills;
- QA scripts;
- documentation.

## 7. Что это не

Проект не является:

- интернет-магазином;
- marketplace interface;
- generic SaaS dashboard;
- обычным landing page;
- PowerPoint clone;
- случайной демонстрацией анимаций;
- одноразовой презентацией.

Не оптимизируй только под один конкретный экран. Думай о повторяемой системе.

## 8. Разрешенная технологическая свобода

В новом проекте разрешены современные подходы, если они помогают цели:

- React;
- TypeScript;
- Vite;

- Tailwind CSS;

4. как она повлияет на mobile performance;
5. как будет обеспечен rollback.

Не добавляй зависимости ради моды.

## 9. Operating modes / режимы работы

Перед началом задачи классифицируй режим. Если пользователь явно указал режим, следуй ему.

### 9.1 Safe operation

Используй для анализа, ревью, планирования и объяснений.

Примеры:

- изучить структуру проекта;
- объяснить, как работает часть проекта;
- найти риски;
- предложить план;
- проверить документацию;
- сделать visual review;
- проверить, доступны ли инструменты.

Правила:

- не редактируй файлы;
- не устанавливай зависимости;
- не создавай коммиты;
- отвечай кратко и практично;
- если действие невозможно в текущей среде, объясни, что пользователю нужно сделать в UI.

### 9.2 Foundation setup

Используй для начальной настройки нового проекта.

Примеры:

- создать документацию;
- создать AGENTS.md;
- создать PROJECT_CONTEXT.md;
- создать QA_CHECKLIST.md;
- создать Codex skills;
- создать docs/BRAND_DNA.md;
- описать reference usage.

Правила:

- не создавай UI-код без отдельного запроса;
- не устанавливай зависимости без отдельного согласования;
- зафиксируй центральную идею проекта;
- отдели stable reference от новой platform direction.

### 9.3 Experiment

Используй для творческих, визуальных и архитектурных экспериментов.

Примеры:

- новая навигация;
- новая motion-концепция;
- новый presentation layout;
- новый animated infographic pattern;
- prototype компонента;
- компонент, созданный с помощью 21st.dev Magic MCP.

Правила:

- работай в отдельной ветке или изолированной папке;
- не ломай stable/reference материалы;
- сначала предложи 2–3 концепции, если задача визуально открытая;
- оцени риски;
- оптимизируй не только красоту, но и переиспользуемость.

### 9.4 Platform evolution

Используй, когда задача влияет на будущую систему генерации презентаций.

Примеры:

- design tokens;
- reusable components;
- content schema;
- presentation template engine;
- chart abstraction;
- responsive layout primitives;
- data-to-presentation pipeline;
- документация по генерации презентаций.

Правила:

- объясни, как изменение помогает повторяемости;
- избегай преждевременного overengineering;
- делай маленькие composable primitives;
- добавляй примеры;
- обновляй документацию.

### 9.5 Targeted fix

Используй для небольших исправлений.

Примеры:

- исправить mobile overflow;
- поправить текст;
- улучшить spacing;
- исправить один баг анимации;
- улучшить контраст.

Правила:

- измени минимум файлов;
- не смешивай несвязанные правки;
- перед изменениями кратко объясни, что будет сделано;
- после изменений дай summary и проверки.

## 10. Default mode

Если режим не указан:

- Safe operation — для анализа, ревью и планирования;
- Foundation setup — для начальной настройки проекта;
- Experiment — для открытых дизайн-задач;
- Platform evolution — для повторяемой архитектуры;
- Targeted fix — для ясных небольших исправлений.

Не делай простые задачи бюрократичными. Для рискованных изменений сначала покажи план.

## 12. Design principles

Сохраняй:

- премиальность;
- точность;
- сильную типографику;
- продуманную сетку;
- ощущение flagship boutique / luxury lifestyle;
- графит, камень, дерево, ivory, теплые amber-акценты;
- фирменные brand assets;
- спокойную уверенность;
- высокую читаемость данных;
- современную motion-культуру.

Избегай:

- marketplace aesthetics;
- скидочных и промо-механик;
- cheap gold luxury;
- cyberpunk blue glow;
- generic SaaS dashboard;
- бессмысленных карточек;
- emoji;
- декоративного шума;
- анимации ради анимации;
- перегруженных экранов.

Разрешено быть смелее старой презентации в композиции, motion, навигации и инфографике, если это не разрушает brand DNA и читаемость бизнес-смысла.

## 13. Business storytelling principles

Не начинай с UI. Начинай с управленческой истории.

Для каждой презентации определи:

- аудиторию;
- бизнес-контекст;
- цель обсуждения;
- core thesis;
- доступные данные;
- ключевую интригу;
- сильнейший инсайт;
- ограничения данных;
- ожидаемое управленческое действие.

Каждый экран должен отвечать:

- что изменилось;
- почему это важно;
- чем это подтверждается;
- какое управленческое действие из этого следует.

## 14. Presentation architecture

Не предполагай фиксированное количество экранов.

Возможные типы секций:

- Executive cover;
- Key result;
- KPI strip;
- Business context;
- Conversion funnel;
- Channel performance;
- Category performance;
- Consultant performance;
- Client journey;
- Before / after;
- Timeline;
- Scenario matrix;
- Risk map;
- Opportunity map;
- Roadmap;
- Decision summary;
- Appendix / data layer.

Каждая секция должна иметь:

- один главный тезис;
- короткое executive-пояснение;
- визуальное доказательство;
- управленческий вывод.

## 15. Responsive success criterion

Успех проекта невозможен без отличного качества на mobile.

Обязательные target environments:

- desktop Chrome;
- desktop Safari;
- iPhone Safari portrait;
- iPhone Safari landscape;
- iPhone Pro Max family начиная с iPhone 14 Pro Max.

Mobile rules:

- нет горизонтального overflow;
- не обрезается критичный текст;
- графики читаемы или трансформируются в mobile-friendly cards;
- fixed navigation не закрывает смысл;
- tap targets удобны;
- motion плавный;
- presentation выглядит специально спроектированной для iPhone, а не просто сжатой desktop-версией.

## 16. Motion principles

Motion должен быть современным, кинематографичным и осмысленным.

Используй motion для:

- направления внимания;
- раскрытия иерархии;
- объяснения изменения данных;
- премиальных переходов между секциями;
- создания ощущения digital experience.

Избегай:

- bounce;
- elastic effects;
- случайных fly-ins;
- слишком активного parallax;
- постоянного фонового движения;
- эффектов, которые мешают читать бизнес-смысл.

Сложная анимация допустима, если она работает на desktop и mobile и не ухудшает восприятие.

## 17. Data visualization principles

Инфографика должна быть красивой, но не декоративной.

Правила:

- сначала insight, потом chart;
- выбирай самый простой график, который объясняет мысль;
- не делай BI-dashboard внутри презентации;
- показывай delta, baseline и comparison period;
- объясняй аномалии;
- отделяй факт, гипотезу и рекомендацию;
- на mobile графики могут превращаться в cards, story blocks или simplified views.

Для ключевых метрик уточняй:

- период;
- базу;
- numerator;
- denominator;
- exclusions;
- create date / complete date / другое основание расчета;
- comparison point.

## 18. 21st.dev / Magic MCP usage

21st.dev Magic MCP можно использовать как ускоритель создания UI-компонентов.

Используй для:

- hero sections;
- KPI cards;
- navigation patterns;
- timeline;
- comparison matrix;
- scenario cards;
- animated business blocks;
- responsive UI patterns.

Не используй как:

- источник brand strategy;
- финальный арт-дирекшн;
- повод для generic SaaS aesthetics;
- повод игнорировать mobile;
- повод добавлять лишние зависимости.

После генерации компонента:

1. проверь, не выглядит ли он generic;
2. убери лишний glow, gradients, noise;
3. приведи к brand tokens;
4. усили типографику и spacing;
5. проверь desktop и iPhone;
6. интегрируй только если компонент reusable;
7. объясни, что принято и что отклонено.

Если MCP недоступен в UI, честно сообщи об этом и предложи ручную реализацию.

## 20. File discipline

Перед изменением файлов:

1. проверь текущее состояние проекта доступными средствами UI;
2. определи точные файлы для изменения;
3. объясни намерение;
4. выбери режим работы;

Не смешивай:

- content rewrite и design tokens;

- reference import и UI implementation.

## 21. Response style for Codex

Для нетривиальных задач отвечай структурно:

1. понимание задачи;

4. краткий план;

## 22. Quality bar

Не достаточно, чтобы “работало”.

Результат должен быть:

- эстетически сильным;
- технически стабильным;
- responsive;
- управленчески убедительным;
- reusable where appropriate;
- без generic AI visual noise;
- достоин senior-аудитории и премиального бренда.

Главный стандарт:

> Brand DNA фиксируем. Форму, motion, композицию и инфографику разрешаем изобретать.
