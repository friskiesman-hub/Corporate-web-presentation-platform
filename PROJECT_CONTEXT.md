# PROJECT_CONTEXT.md

## 1. Назначение проекта

Проект — экспериментальная платформа для создания премиальных корпоративных web-презентаций в фирменном стиле BORK.

Цель — не одна презентация, а система, которая позволит быстро создавать управленческие web-презентации на основе бизнес-данных: продажи, маркетинг, e-commerce, CRM, категории, доставка, оплата, клиентский опыт, эффективность персональных консультантов и digital-каналов.

Проект должен объединить:

- executive storytelling;
- premium visual design;
- modern motion;
- responsive layout;
- animated infographics;
- data-driven sections;
- reusable presentation components;
- brand DNA BORK.

## 2. Новая стратегическая рамка

Мы не продолжаем старую 5-слайдовую презентацию как техническую основу.

Мы создаем новый проект, где предыдущая стабильная презентация BORK Internet Boutique Q2 2026 используется как visual reference и источник lessons learned.

Старая версия не должна ограничивать:

- технологический стек;
- количество экранов;
- архитектуру;
- уровень креатива;
- тип навигации;
- уровень анимации;
- способ работы с данными;
- формат генерации новых презентаций.

Ограничения сохраняются только на уровне brand DNA:

- премиальность;
- цветовая палитра;
- лого;
- паттерн;
- герб / фирменные элементы;
- отсутствие marketplace aesthetics;
- высокое качество desktop и iPhone.

## 3. Целевая аудитория презентаций

Основная аудитория будущих презентаций:

- генеральный директор;
- коммерческий директор;
- директор по продажам интернет-бутика;
- руководитель performance marketing;
- руководители розницы;
- руководители клиентского сервиса;
- лидеры групп;
- участники стратегических и операционных совещаний.

Презентации должны быть достаточно красивыми для демонстрации senior-аудитории и достаточно точными для управленческих решений.

## 4. Тематический фокус первой версии платформы

На первом этапе ограничиваемся темами:

- Internet Boutique performance;
- sales review;
- marketing / promotion review;
- channel efficiency;
- conversion and funnel analysis;
- category dynamics;
- personal consultant performance;
- customer journey;
- operational risks;
- Q-focus / roadmap.

Позже платформа может расшириться на другие бизнес-направления.

## 5. Желаемый пользовательский сценарий будущей платформы

Идеальный сценарий:

1. Пользователь описывает тему презентации.
2. Пользователь передает данные или краткое описание результатов.
3. Система определяет narrative structure.
4. Система предлагает набор секций.
5. Система выбирает подходящие infographic patterns.
6. Система генерирует web-презентацию в фирменном стиле.
7. Пользователь вносит правки в тезисы, данные или тональность.
8. Система проверяет desktop/mobile качество.
9. Готовая презентация публикуется или экспортируется.

## 6. Product principles

Презентация должна быть:

- business-first: сначала смысл, потом UI;
- premium: дорогая, спокойная, точная;
- cinematic: modern motion, но без шума;
- data-literate: метрики понятны и честны;
- responsive: desktop и iPhone равнозначны;
- reusable: хорошие решения должны становиться компонентами;
- brand-coded: цвет, паттерн, лого и фирменная эстетика должны быть узнаваемы.

## 7. Возможная архитектура проекта

Архитектура может развиваться постепенно.

Потенциальная структура:

```text
src/
  components/
  sections/
  presentations/
  data/
  tokens/
  motion/
  charts/
  templates/
  lib/
public/
  brand/
  references/
docs/
  BRAND_DNA.md
  REFERENCE_USAGE.md
  PLATFORM_ROADMAP.md  # potential future file, not required now
.agents/
  skills/
```

Не нужно создавать всю структуру сразу без необходимости. Но новые решения должны учитывать будущую повторяемость.
`docs/PLATFORM_ROADMAP.md` указан как возможный будущий документ, а не как обязательная часть текущей структуры.

## 8. Рекомендуемый стартовый стек

Для первого технологического прототипа рекомендуется рассмотреть:

- Vite;
- React;
- TypeScript;
- Tailwind CSS;
- Framer Motion;
- позже — chart library после выбора первых infographic patterns.

Next.js не является обязательным на старте. Он может понадобиться позже, если появятся:

- routing для множества презентаций;
- серверная генерация;
- авторизация;
- CMS;
- публикация отдельных презентаций как страниц.

## 9. Роль 21st.dev / Magic MCP

21st.dev Magic MCP используется как инструмент ускоренного создания отдельных UI-компонентов.

Он не определяет финальный стиль проекта.

Допустимые сценарии:

- сгенерировать 2–3 варианта KPI cards;
- сгенерировать responsive timeline;
- сгенерировать comparison matrix;
- сгенерировать navigation idea;
- сгенерировать animated hero pattern.

После генерации компонент обязательно адаптируется под brand DNA и проходит premium UI review.

## 10. Роль предыдущей презентации

Предыдущая стабильная презентация должна быть помещена в проект как reference material или приложена в Codex UI как контекст.

Она отвечает на вопрос:

> Какое ощущение премиальности и brand atmosphere уже было найдено?

Она не отвечает на вопрос:

> Какой должна быть архитектура новой платформы?

## 11. Критерии успеха первой фазы

Первая фаза успешна, если:

- создана новая рамка проекта;
- stable reference не ограничивает архитектуру;
- определен brand DNA;
- создан набор Codex skills;
- выбран технологический подход;
- создан первый reusable prototype section;
- prototype выглядит отлично на desktop и iPhone Pro Max;
- проект способен принять новые бизнес-данные и превратить их в презентационный narrative.

## 12. Главная формула проекта

> Мы сохраняем ДНК бренда, но изобретаем новый формат корпоративной web-презентации: кинематографичный, мобильный, интерактивный, data-driven и достойный senior-аудитории.
